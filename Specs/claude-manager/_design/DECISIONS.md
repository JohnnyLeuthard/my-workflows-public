# Claude Manager — Decision Log

Append-only. Each entry: what was decided, why, and what was rejected. The spec
(`SPEC.md`) records outcomes; this file records reasoning. Never rewrite old
entries — supersede them with new ones.

Entries D1–D10 were **reconstructed 2026-07-11** from git history, README/TASKS content, and
the code as built (the project predates this `_design/` folder). D11 onward were written as
they happened. Security-review entries (D11–D22) have a fuller paper trail in
`../SECURITY-REVIEW-PLAN.md`.

---

## D1. Zero npm dependencies, forever

**Why**: The repo lives on a portable cloud-synced drive used across multiple machines —
`git clone` + `node scripts/…` must be the entire setup on any of them. Past experience:
silent installs left 500MB+ messes in shared workflows. This is also why CLAUDE.md forbids
any install without explicit per-request approval.
**Rejected**: chalk/commander/express and friends (each trivially replaced by ~20 lines);
any build step; TUI frameworks.

## D2. Docs first, tools second (phased roadmap)

**Why**: The core value is *understanding* `~/.claude` — a reference explaining what each
folder is and whether it's safe to delete is useful with no code at all. Tools automate what
the docs already justify. Hence Phase 1 README reference → Phase 2 scripts → Phase 3 UI.
**Rejected**: Leading with a cleanup tool (dangerous without the knowledge base; the
knowledge base *is* the product).

## D3. All persistent context lives in the repo, never in `~/.claude`

**Why**: Anything written to `~/.claude` (including Claude Code's auto-memory) is
machine-local and invisible on the other computers this drive plugs into — and this project's
whole subject is keeping `~/.claude` clean. Memory-worthy facts go in CLAUDE.md.
**Rejected**: The harness auto-memory system; per-machine notes; temp files outside the repo.

## D4. Read-only scanners; deletion is a separate, dry-run-by-default tool

**Why**: Safety first — scan.js/audit.js can be run by anyone with zero risk, which is what
makes people willing to run them. Destructive action requires an explicit choice
(`clean.js --execute`), not an accidental omission.
**Rejected**: A `--delete` flag on scan.js; interactive delete prompts inside the dashboard.

## D5. Dual output: ANSI terminal by default, HTML on request

**Why**: Terminal output pipes and greps (`--no-color` for logs); HTML communicates to humans
(cards, size bars, clickable paths). Same data, two renderers, one scanner.
**Rejected**: HTML-only (breaks piping/CI); TUI framework (violates D1); separate tools per
output.

## D6. `--html` serves via a local server; `--html-only` writes a static file

**Why**: "Open this folder in Finder" from a browser page requires a native call — a
`file://` page can't do it, so a tiny localhost server handles `/open` (and the dashboard's
Refresh via `/rescan`). The static variant exists for saving/sharing, where links degrade to
clipboard-copy. 10-minute auto-shutdown so no forgotten server lingers.
**Rejected**: `file://` links (browsers block folder-opening); Electron/webview (D1); leaving
the server running until killed.

## D7. Hardcoded `FOLDER_METADATA` knowledge base + loud UNKNOWN flagging

**Why**: The descriptions, importance levels, and pros/cons *are* the curated product —
they change only when Claude Code changes, so they belong in code where they version with
the repo. Anything on disk but not in the list renders as UNKNOWN with a
do-not-delete-without-research warning, which both protects the user and generates the
to-document backlog.
**Rejected**: External JSON/YAML config (nothing else consumes it; one more file to drift);
guessing importance for unknown folders.

## D8. Disclaimer-first security posture

**Why**: audit.js is pattern matching, not a security review. A clean result must never read
as "you're safe" — false negatives are certain, false positives happen, and the user owns
due diligence. So the disclaimer renders unconditionally in both outputs, and OK findings say
"no known patterns matched," never "clean." (Added in dedicated commits after the wording was
initially softer; strengthened to cover false negatives explicitly.)
**Rejected**: Disclaimer only on findings; only in docs; confidence language ("your folder is
secure ✓").

## D9. Fixture with fake credentials, examples generated then sanitized

**Why**: A detector nobody can exercise is a detector nobody trusts —
`demo-shell-snapshot.sh` gives a one-copy test that lights up every pattern, with obviously
fake values and a header listing expected findings. Example HTML is generated by *running the
real script* against a staged fake HOME (so it never drifts from real output), then
sanitized (`/Users/yourname`, EXAMPLE banner, dead JS stripped).
**Rejected**: Hand-maintained example HTML (drifts, and did); testing against the author's
real `~/.claude` (real data in a tracked file is exactly the failure mode this project
warns about).

## D10. `reports/` is gitignored, and `.gitignore` blocks real `~/.claude` data wholesale

**Why**: Generated reports describe the local machine's real data and must never be
committed; `.gitkeep` keeps the folder. Belt-and-suspenders patterns (`*.jsonl`,
`backups/`, `projects/`, `shell-snapshots/`, …) make accidentally committing copied
`~/.claude` content hard even by mistake.
**Rejected**: Committing "one harmless dashboard" (it contains real usernames/paths/sizes).

*D11–D22 implement the 2026-07-11 security/design review (`../SECURITY-REVIEW-PLAN.md`,
items S1–S8, D1–D6 there). Findings were proposed first, approved by the human, then
implemented in two passes.*

## D11. Audit coverage: five scanners, and honest disclosure of what isn't scanned

**Why** (review S4): the audit originally scanned three targets while the README called
`projects/` the most sensitive folder — a green result quietly overpromised. Added top-level
`*.json` secret sweep and a POSIX permission check (`chmod 600` action) for
`.credentials.json`/`settings.json`, plus an always-rendered "Not scanned" section. The
permission check paid for itself immediately: it flagged the author's own `settings.json` at
644 on first real run.
**Rejected**: Scanning `projects/` transcripts by default (slow, false-positive-heavy —
deferred to an opt-in `--deep`, see NEXT-STEPS.md); pretending coverage by listing targets
without the not-scanned list.

## D12. `.credentials.json`: presence reported, contents never scanned

**Why**: It is *supposed* to contain auth tokens — pattern-scanning it yields only noise and
a scary HIGH for normal operation. Its risks are permissions (D11) and copies leaving
`~/.claude`, so that's what the finding says.
**Rejected**: Scanning it (guaranteed false alarm); ignoring it entirely (users should know
it exists and what it is).

## D13. HTML reports withhold secret values entirely; terminal keeps 6-char previews

**Why** (review S5): this repo lives under OneDrive — a saved report containing even
truncated secret prefixes syncs evidence of leaks to a third party. Terminal output isn't
persisted, so it keeps the truncated preview (never full values, in either output). Docs
carry a "Where reports go" cloud-sync warning.
**Rejected**: Truncated previews in HTML (still a fragment on someone else's server);
dropping previews everywhere (terminal preview is how you find the actual line).

## D14. Local report server: per-run token + Host check + normalized path containment

**Why** (review S2/S3): binding to 127.0.0.1 does not stop JS on any website from firing
requests at localhost ports for the 10 minutes the server runs. A random per-run token
gates every route (including `/`, since the report names flagged files); the Host check kills
DNS rebinding; `/open` containment uses `path.relative` after `resolve` (the old
`startsWith` passed `~/.claude/../../etc` and `~/.claude-evil`) and rejects leading-`-`
paths. Impact was modest (opening Finder windows) but this is a security tool — it models
the right pattern.
**Rejected**: Trusting the loopback bind; CORS headers (wrong tool — no credentialed
cross-origin reads to block); POST-only (doesn't stop cross-origin sends).

## D15. Secret patterns: most-specific first, de-duplicated by match offset

**Why** (review S1/S7 + D5): the original `sk-[a-zA-Z0-9]{20,}` could not match a real
Anthropic key (`sk-ant-api03-…` — the dash ends the match at 3 chars): the headline detector
of a `~/.claude` audit tool missed Claude's own key format. Split Anthropic/OpenAI patterns
and added GitHub OAuth/app, Slack, Google, npm, JWT, and private-key-block detectors. One
span matching several patterns now reports once — first (most specific) pattern wins by
start-offset dedup. A `NAME=value` line still yields two findings (name match + value match
at different offsets), which is correct and documented in the fixture.
**Rejected**: One mega-regex (unreadable, undebuggable); cross-file or cross-line dedup
(hides real distinct hits); entropy scoring (false-positive machine, needs tuning this tool
can't justify).

## D16. Exit codes: 2 = HIGH, 1 = WARN, 0 = clean

**Why** (review S6): scripts and CI can't read prose — `audit.js && deploy` must fail when
credentials are found. Set via `process.exitCode` (not `process.exit()`) so `--html` server
mode still serves; the code applies when the process actually exits.
**Rejected**: Always-0 (status quo — useless for automation); failing on WARN alone with no
distinct code (loses information).

## D17. Read caps: skip files >5 MB or with NUL in the first 512 bytes

**Why** (review S8): `readFileSync` into a regex on unbounded input means a stray multi-GB
file stalls or OOMs the audit. Secrets live in small text files; oversized/binary files get a
WARN telling the user to inspect manually — visible, not silent.
**Rejected**: Streaming scans (complexity unjustified at these sizes); silent skip (coverage
lie).

## D18. Shared `scripts/lib/common.js` instead of two copies of everything

**Why** (review D1): ~40% of the two scripts was duplicated — ANSI, arg parsing, escaping,
and the *entire server*, which is why the S2/S3 fixes would otherwise have been written
twice. One `serveReport({ html, claudeDir, label, onRescan })` serves both; scan.js passes
an `onRescan` callback, audit.js doesn't. Plain CommonJS `require('./lib/common')` keeps
direct `node scripts/…` invocation working.
**Rejected**: npm workspace/private package (D1 — the constraint, not the finding); keeping
copies in sync by hand (they had already drifted).

## D19. Folder sizes: one pure-JS walk; `du` dropped

**Why** (review D3): `du -sk` on macOS reports allocated blocks, `du -sb` on Linux apparent
bytes — same folder, different numbers per OS, plus two subprocess spawns per folder. The
recursive `lstat` walk is consistent everywhere and fast enough at `~/.claude` scale
(tens of MB).
**Rejected**: Keeping `du` with a documented discrepancy (a dashboard whose numbers change by
OS erodes trust in everything else it says).

## D20. Freeable percentages are labeled heuristics

**Why** (review D4): `freeablePct` values are hand-estimated editorial guesses, but the
dashboard rendered "Est. freeable: ~20.4 MB" like a measurement. Both renderers now label it
("hand-estimated per folder, not a measurement"); the metadata carries the same caveat in a
comment. Real measurement is clean.js's job (mtime-based), not the scanner's.
**Rejected**: Building mtime measurement into scan.js now (scope creep into exactly the
logic clean.js needs to own); removing the estimate (it's genuinely useful triage).

## D21. `package.json` with zero dependencies

**Why** (review D6): gives `npm run scan`/`npm run audit` ergonomics, an explicit
`engines.node >= 16` floor, and `private: true` — while installing nothing (creating the
file runs no installer, honoring the no-install rule). Empty `dependencies`/`devDependencies`
are declared explicitly so any PR adding one is visible in review.
**Rejected**: No package.json (was fine, but the Node floor lived only in prose); adding
even a test-runner devDependency (D1).

## D22. Security review workflow: findings ledger → human approval → implementation passes

**Why**: The review itself produced `SECURITY-REVIEW-PLAN.md` — findings with file:line,
fix guidance, and acceptance criteria, written so any model could implement them. Nothing
was changed until the human approved; S4/S5 shipped first on request, the rest on a second
go-ahead; the ledger's status column tracks what landed when. This is the pattern for future
reviews: propose in a ledger, let the human pick, implement in batches, verify per the
acceptance criteria.
**Rejected**: Fix-as-you-find (mixes judgment calls into unreviewed diffs); filing issues
only (loses the implementation-ready detail).

## D23. `_design/` folder: this project adopts the workspace design-seed convention *(2026-07-11)*

**Why**: Other projects in this workspace (`debate-lab`, `research-lab`, `_ICM-Template`)
keep a `_design/` folder that is both design record and drop-in build seed — spec for
outcomes, decision log for reasoning, router, next-steps. Adopting it here makes the
convention uniform and replaces the stale root `SPEC.md`, which mirrored implementation
detail (CSS class inventories) and had already drifted from the code twice. The root file
was removed; its durable content was absorbed into `_design/SPEC.md` at behavioral altitude.
**Rejected**: Keeping both specs (two sources of truth, one already wrong); updating the
root SPEC.md in place (implementation mirrors always drift — the code is the implementation
record; the spec should hold the contract).
